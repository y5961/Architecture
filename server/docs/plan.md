# תוכנית מגירה: מעבר מ-API מונוליתי לארכיטקטורת מיקרו-שירותים (פירוט לפי ישויות אמיתיות)

מסמך זה מפרט מפת דרכים תאורטית למעבר של הפרויקט לארכיטקטורת Microservices, עם חלוקה ישירה של הישויות (Entities) לשירותים ייעודיים, בהתאם לניתוח תיקיות Models ו-Data.

---

## 1. פירוק לשירותים (Service Decomposition)

### שירות משתמשים וזהות (User/Identity Service)
- **ישויות:** User, Card
- **למה:** ניהול משתמשים, הרשאות, התחברות, רישום, שמירת פרטי כרטיסי אשראי. שירות זה אחראי על אימות והרשאה, ניהול פרטי משתמשים וכרטיסים.

### שירות תורמים (Donor Service)
- **ישויות:** Donor
- **למה:** ניהול תורמים, פרטי תרומה, היסטוריית תרומות. שירות זה מרכז את כל המידע והפעולות הקשורות לתורמים.

### שירות מכרזים ופרסים (Auction & Prize Service)
- **ישויות:** Auction, Prize, AuctionPrize
- **למה:** ניהול מכרזים, שיוך פרסים למכרזים, עדכון סטטוס מכרזים. שירות זה אחראי על פתיחה/סגירה של מכרזים, ניהול פרסים, ושיוך פרסים למכרזים.

### שירות הזמנות ורכישות (Order Service)
- **ישויות:** Order, OrdersGift, OrdersPackage, Gift, Package
- **למה:** ניהול הזמנות, שיוך מתנות וחבילות להזמנה, סטטוס הזמנה. שירות זה מרכז את כל תהליך הרכישה וההזמנה.

### שירות תשלומים (Payment Service)
- **ישויות:** (בהתאם למימוש עתידי, לדוג' Transaction, Payment)
- **למה:** עיבוד תשלומים, חיובים, קבלות. שירות זה יטפל באינטגרציה עם ספקי סליקה וניהול עסקאות.

### שירות התראות (Notification Service)
- **ישויות:** (בהתאם למימוש עתידי, לדוג' Notification)
- **למה:** שליחת מיילים, SMS, פוש. שירות זה יטפל בניהול תבניות ושליחת התראות.

---

## 2. תשתיות (Infrastructure Plan)

- **Docker:** כל שירות ירוץ כ-Container עצמאי.
- **API Gateway (YARP):** שער מרכזי לניהול ניתוב, אבטחה, ואגרגציה של קריאות API.
- **Message Broker (RabbitMQ/Kafka):** תקשורת אסינכרונית בין שירותים (Events, Commands).
- **Orchestration (Kubernetes/Azure Container Apps):** ניהול פריסה, סקיילינג, ניטור.

---

## 3. אסטרטגיית נתונים (Data Strategy)

- **Database-per-Service:** לכל שירות מסד נתונים עצמאי (למשל, SQL Server, PostgreSQL, MongoDB), עם טבלאות רק לישויות שבתחום השירות.
- **מעבר הדרגתי:** מיגרציה של נתונים מה-DB המשותף, כולל כתיבת סקריפטים להעברת ישויות רלוונטיות לכל שירות.
- **אינטגרציה:** העברת מידע בין שירותים באמצעות API או Messaging בלבד, ללא גישה ישירה ל-DB של שירות אחר.

---

## 4. Shared Kernel (שיתוף קוד)

- **חבילת NuGet משותפת:** ספריה משותפת שתכיל:
  - DTOs (Data Transfer Objects) משותפים (למשל, UserDTO, OrderDTO, PrizeDTO).
  - קונטרקטים לאירועים (Event Contracts).
  - Utilities ולוגיקה עסקית כללית.
- **ניהול גרסאות:** שמירה על תאימות לאחור ותיעוד שינויים.

---

## 5. מפת דרכים שלבים (Phase-by-Phase Roadmap)

1. **מיפוי גבולות שירותים וישויות:** ניתוח הקוד הקיים, שיוך כל Entity לשירות המתאים.
2. **הפרדת שירות משתמשים (User/Card):** הוצאת ניהול משתמשים וכרטיסים לשירות עצמאי ראשון.
3. **הקמת API Gateway:** פריסת YARP וניתוב קריאות לשירותים.
4. **הפרדת שירותי Donor, Auction, Prize:** הוצאת לוגיקת תורמים, מכרזים ופרסים לשירותים נפרדים.
5. **הפרדת שירות Order (הזמנות):** הוצאת ניהול הזמנות, מתנות וחבילות לשירות עצמאי.
6. **הטמעת Message Broker:** מעבר לתקשורת אסינכרונית בין שירותים.
7. **הפרדת מסדי נתונים:** מעבר הדרגתי ל-Database-per-Service, כולל מיגרציה של נתונים.
8. **הקמת Shared Kernel:** פיתוח והפצה של חבילת NuGet משותפת.
9. **הפרדת שירותים נוספים (Payment, Notification):** לפי הצורך והעדיפות העסקית.
10. **בדיקות, ניטור ואופטימיזציה:** בדיקות אינטגרציה, ניטור ביצועים, שיפור תהליכים.

---

## הערות

- כל שלב ילווה בבדיקות רגרסיה, ניטור ותיעוד.
- יש להבטיח תאימות לאחור (Backward Compatibility) לאורך כל התהליך.
- מומלץ להתחיל בסביבה לא-פרודקשן ולבצע מעבר הדרגתי.
